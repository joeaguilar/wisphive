## TUI Enhancements

### AskUserQuestion Detail View (`c1d0373`)
- Renders question content (header, question text, numbered options) for AskUserQuestion events
- Users answer directly from the TUI: number keys select options, `O` opens text input for custom response
- Answers passed back to Claude Code via `updatedInput` on PermissionRequest hook response
- Key discovery: AskUserQuestion must be handled via **PermissionRequest** (not PreToolUse) — PreToolUse hooks cannot pass answers back

### ExitPlanMode Detail View (`33b8c53`)
- Extracts plan content from Claude Code's transcript JSONL file (`transcript_path`)
- Reads the last assistant text message from the transcript and passes it as `event_data.plan_content`
- Dedicated plan detail renderer with accept/reject/reject-with-feedback keybindings

### Markdown Preview Toggle (`1270c64`)
- Press `P` in any detail view to toggle between raw text and rendered markdown
- Uses `pulldown-cmark` to render headers, bold, italic, code blocks, lists with styled ratatui Spans
- Status bar shows `[P]review` / `[P]raw` toggle indicator

### Detail View Modal Rendering Fix
- **Root cause found:** `draw_detail_view` never called `draw_modal` — modals opened from the detail view (deny-with-message, edit-input, approve-with-context) were created in app state but never rendered visually
- This bug affected ALL modal-based actions in ALL detail views, not just AskUserQuestion
- Hours of debugging narrowed it down: keys were reaching the correct handler, modals were being set, but the rendering call was missing

### History Reimport Fix (`f99264e`, closes #39)
- The daemon's `event_ingest.rs` was seeking to EOF on startup, skipping all auto-approved events from `events.jsonl` written before the daemon started
- Added on-demand reimport: `ReimportEvents` protocol message triggers full `events.jsonl` re-read with deduplication via unique index on `tool_use_id`
- TUI sends `ReimportEvents` before `QueryHistory` when entering the history view

## Web UI (wisphive_web crate — from zero to feature parity)

### Architecture
```
Browser ←WebSocket→ wisphive_web (axum) ←Unix socket→ wisphive daemon
```
- **Backend:** Axum server with WebSocket bridge that translates between browser WebSocket and daemon Unix socket. Same JSONL protocol — zero translation needed.
- **Frontend:** React + TypeScript + Vite. `useWisphive()` hook manages WebSocket connection, auto-reconnect, and typed protocol messages.
- **Deployment:** `rust-embed` embeds the built frontend into the binary. Single binary, no runtime deps. `wisphive web` just works.
- **Development:** `wisphive web --dev` serves only the WebSocket; Vite dev server handles frontend with HMR. Vite proxies `/ws` to the backend.

### CLI Subcommand
```bash
wisphive web                        # Production: http://127.0.0.1:3100
wisphive web --port 8080            # Custom port
wisphive web --host 0.0.0.0         # LAN access (shows local IP)
wisphive web --dev                  # Dev mode (Vite for frontend)
```

### Design Decisions
- **CSS Design Tokens:** All sizes, spacing, typography, and touch targets are CSS variables. Responsive breakpoints override `:root` values so all components scale automatically.
- **Mobile-first responsive:** Sidebar collapses to top nav, queue/detail stack vertically, 44px touch targets, 16px input font (prevents iOS zoom), modals fill 95% viewport on mobile.
- **No extra dependencies for rendering:** Markdown rendered via regex-based HTML conversion, diffs via CSS-colored line-by-line display. Keeps bundle small (~220KB).
- **REST for config, WebSocket for everything else:** Config read/write is a simple filesystem operation — no need to pipe through the daemon protocol.

## Key Learnings (saved to memory)

1. **AskUserQuestion hooks:** Must use PermissionRequest (not PreToolUse). PreToolUse cannot pass answers back via `updatedInput`.
2. **Detail view modals:** Every TUI view must call `draw_modal` for modal overlay rendering. Missing this causes invisible modals.
3. **Mobile responsive CSS:** Always ship responsive design from the start. Never treat it as a follow-up.
4. **Hook response schema:** Always verify against Claude Code's actual schema — never guess field values.
