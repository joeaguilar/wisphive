# Open-Source Readiness: Wisphive

Research findings for bringing Wisphive into the open-source sphere. This document covers licensing, code hygiene, community scaffolding, and distribution concerns.

---

## 1. Licensing

### Current State
- `Cargo.toml` declares `license = "MIT"`, inherited by all workspace crates.
- **No `LICENSE` file exists at the repo root.** This is the single most important gap: without the full license text, the Cargo.toml field is a declaration of intent but not legally enforceable in all jurisdictions.

### What To Do
- [ ] Add a `LICENSE` file at the repo root containing the full MIT license text with copyright line: `Copyright (c) 2024 Josef Aguilar`
- [ ] Optionally consider dual-licensing as `MIT OR Apache-2.0` (the Rust ecosystem standard). This provides patent protection that MIT alone lacks. If chosen, add both `LICENSE-MIT` and `LICENSE-APACHE` files and update `Cargo.toml` to `license = "MIT OR Apache-2.0"`.

### Recommendation
**MIT is fine** for a standalone tool like Wisphive. Dual-licensing is nice-to-have but not required. The critical action is adding the actual LICENSE file.

## 6. Code Signing & Distribution

### Current State
`install.sh` uses **ad-hoc code signing** (`codesign -s -`), which is a self-signature that satisfies macOS Gatekeeper for locally-built binaries. This is the right approach for source-distributed software.

### Do You Need Apple Developer Code Signing?
**No**, unless you plan to distribute pre-built binaries via download (not source). If you only distribute source (users clone + `./install.sh`), ad-hoc signing is sufficient.

### If You Want Pre-Built Binary Releases
- Consider GitHub Releases with binaries built by CI (cross-compile for macOS arm64/x86_64, Linux x86_64)
- macOS binaries from CI will be ad-hoc signed (no Gatekeeper bypass for downloaded binaries — users must `xattr -d com.apple.quarantine` or right-click > Open)
- A paid Apple Developer account ($99/year) + notarization would bypass Gatekeeper entirely, but is overkill for an early open-source project
- Homebrew tap is the cleanest macOS distribution path (builds from source, no signing needed)

### Crates.io
- [ ] Decide if you want to publish to crates.io. The workspace is ready (`license`, `version`, `authors` set). You'd need to add `description`, `repository`, and `homepage` fields.

## 8. What You Do NOT Need

- **No code signing certificate** — ad-hoc is fine for source distribution
- **No CLA (Contributor License Agreement)** — MIT is simple enough that a CLA is unnecessary for a project this size
- **No license header in every source file** — MIT doesn't require per-file headers (Apache-2.0 recommends them, but you're using MIT)
- **No dependency vendoring** — Cargo.lock is committed (correct for binaries), and all deps are permissive
